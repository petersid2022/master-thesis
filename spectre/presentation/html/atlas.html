Autoregressive vs speculative (inside run())

Both paths share start() through decode(). The fork is the first statement in run(): if -dft was passed, the speculative branch runs; otherwise the infinite AR loop uses the full-prompt batch from decode().

start() pipeline

start() calls initialize → tokenize → decode → run in that order. Only run() diverges; decode() always builds the initial full-prompt batch for the target, and additionally allocates draft samplers and speculation_batch_* when speculation is enabled.

Target prefix before the speculative loop

In run() (spec branch), prompt_tgt still holds the full tokenized prompt. One target forward pass ingests indices 0 … N−2; the final prompt token is held out as last_token so every later step extends the same sequence the AR loop would have produced one token at a time.

Draft model: sliding prefix window

Each draft() call clears draft memory and replays only a suffix of prompt_tgt that fits in draft_context − n_max slots, so there is room for last_token plus up to n_max draft steps.

From draft IDs to the target verification batch

The draft model only returns token ids. The target never sees draft logits; it receives those ids as hypotheses in speculation_batch_tgt at the correct absolute positions, then compares its own samples to the draft column-wise in sample_and_accept.

One speculative round (matches run() loop body)

Order and variable names follow the spec branch of run(): n_past is local each iteration; it is updated before the commit loop, then KV rollback uses that value.

Data flow: draft context, target batch, committed text

prompt_tgt stores the agreed prefix; last_token is always the next symbol to extend. The inner loop pushes the previous last_token into the vector before adopting each element of accepted.

sample_and_accept

Greedy comparison of draft vs. target sample at each logits row; early exit on mismatch; bonus sample if all match.

Target verification batch positions

Let n0 be n_past right before add_batch(last_token, …). After that call the code does n_past += 1, then adds each draft token at n_past + i, which is n0 + 1 + i on the wire.

Verify-batch slot layout

Each appended slot carries token id, absolute position, sequence id, and whether this row should produce logits.

KV cache rollback

After committing accepted tokens, unverified suffix positions are dropped from the target KV cache.
